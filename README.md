## Описание проекта

**MiniCompiler** — это учебный компилятор для упрощённого C-подобного языка, разрабатываемый итеративно с использованием методологии спринтов. Проект создан с целью изучения принципов построения компиляторов: от лексического анализа до генерации кода.

Каждый спринт добавляет новый этап компиляции, строго следуя техническим требованиям и поддерживая чистую архитектуру кода.

**Текущий спринт:** Спринт 1 – Лексический анализатор (лексер).

## Лексический анализ

### Нововведения и изменения

В результате выполнения Спринта 1 в проект были внесены следующие изменения:

1.  **Создана базовая структура проекта:** Организована модульная архитектура с разделением на `src/`, `tests/`, `examples/`, `docs/`.
2.  **Реализована система сборки:** Добавлен `pyproject.toml` для установки пакета и создания CLI-команды `compiler`.
3.  **Разработана спецификация языка:** В `docs/language_spec.md` формально описана лексическая грамматика (токены, ключевые слова, литералы, операторы, комментарии) с использованием EBNF.
4.  **Написан лексический анализатор (сканер):**
    - Преобразует исходный код в последовательность токенов.
    - Отслеживает позицию каждого токена (строка, колонка).
    - Поддерживает все типы токенов из спецификации.
    - Корректно обрабатывает однострочные (`//`) и многострочные (`/* ... */`) комментарии.
    - Распознаёт многосимвольные операторы (`==`, `!=`, `<=`, `>=`, `&&`, `||`, `+=` и др.).
    - Извлекает значения литералов (целые числа, числа с плавающей точкой, строки).
5.  **Добавлена система обработки ошибок:**
    - Обнаруживает недопустимые символы, незакрытые строки и комментарии.
    - Выдаёт понятные сообщения об ошибках с указанием строки и колонки.
    - Реализовано восстановление после ошибок (сканирование продолжается).
6.  **Создан набор тестов:**
    - Автоматизированный раннер тестов (`test_runner.py`).
    - Более 20 тестов на валидные конструкции и 10+ тестов на ошибочные ситуации.
    - Тесты сравнивают фактический вывод лексера с эталонными файлами (`.expected`).
7.  **Добавлен интерфейс командной строки:** Команда `compiler lex` для запуска лексера на файле.

### Спецификация языка

Полная спецификация лексической грамматики находится в файле [`docs/language_spec.md`](docs/language_spec.md). Краткое описание:

- **Ключевые слова:** `if`, `else`, `while`, `for`, `int`, `float`, `bool`, `return`, `true`, `false`, `void`, `struct`, `fn`
- **Идентификаторы:** Начинаются с буквы или `_`, далее буквы, цифры или `_`. Длина до 255 символов.
- **Литералы:** целые (`123`), вещественные (`123.456`), строковые (`"hello"`), булевы (`true`/`false`).
- **Операторы:** арифметические (`+`, `-`, `*`, `/`, `%`), сравнения (`==`, `!=`, `<`, `<=`, `>`, `>=`), логические (`&&`, `||`, `!`), присваивания (`=`, `+=`, `-=`, `*=`, `/=`, `%=`).
- **Разделители:** `( ) { } [ ] ; , .`
- **Комментарии:** однострочные (`// ...`) и многострочные (`/* ... */`).
- **Пробельные символы:** пробел, табуляция, перевод строки (`\n`), возврат каретки (`\r`) — игнорируются.

### Функциональность

Текущая версия программы (`compiler lex`) выполняет следующие функции:

- Читает исходный файл с кодировкой UTF-8.
- Разбивает текст на токены в соответствии с лексической грамматикой.
- Для каждого токена выводит: строку, колонку, тип токена, лексему и (для литералов) значение.
- При обнаружении ошибок выводит диагностические сообщения в `stderr` и завершается с ненулевым кодом возврата.

Проект использует стандартный инструментарий Python. Для "сборки" (установки пакета в режиме разработки) выполните в корневой директории проекта:

Эта команда установит пакет minicompiler и создаст глобально доступную команду compiler. Все изменения в исходном коде будут сразу доступны без переустановки.

Использование
## Запуск лексера
Для запуска лексического анализатора на файле с исходным кодом используйте команду:

```

bash

compiler lex --input <путь_к_входному_файлу> --output <путь_к_выходному_файлу>
Параметры:

--input, -i — обязательный параметр, путь к файлу с исходным кодом (например, examples/hello.src).

--output, -o — обязательный параметр, путь к файлу, куда будут записаны токены (например, tokens.txt).
```
Пример:

```

bash

 python -m src.lexer.cli lex --input examples/hello.src --output tokens.txt             

```

Формат вывода
Выходной файл содержит по одному токену на строку в следующем формате:

````
<СТРОКА>:<КОЛОНКА> <ТИП_ТОКЕНА> "<ЛЕКСЕМА>" [<ЗНАЧЕНИЕ_ЛИТЕРАЛА>]
<СТРОКА> и <КОЛОНКА> — позиция начала токена (индексация с 1).

<ТИП_ТОКЕНА> — имя элемента перечисления TokenType (например, KW_IF, IDENTIFIER, INT_LITERAL).

<ЛЕКСЕМА> — исходный текст токена, заключённый в двойные кавычки.

[<ЗНАЧЕНИЕ_ЛИТЕРАЛА>] — необязательное поле. Присутствует только для токенов-литералов (INT_LITERAL, FLOAT_LITERAL, STRING_LITERAL) и содержит преобразованное значение (число или строку без кавычек).
````
Пример строки вывода:
````
text
2:18 INT_LITERAL "42" 42
В конце файла всегда присутствует токен END_OF_FILE:

text
5:1 END_OF_FILE ""
Примеры
Входной файл examples/hello.src:
text
fn main() {
    int counter = 42;
    return counter;
}
````

Команда:

````
bash
compiler lex -i examples/hello.src -o tokens.txt
````

Содержимое tokens.txt:

````
text
1:1 KW_FN "fn"
1:4 IDENTIFIER "main"
1:8 LPAREN "("
1:9 RPAREN ")"
1:10 LBRACE "{"
2:5 KW_INT "int"
2:9 IDENTIFIER "counter"
2:16 ASSIGN "="
2:18 INT_LITERAL "42" 42
2:19 SEMICOLON ";"
3:5 KW_RETURN "return"
3:12 IDENTIFIER "counter"
3:19 SEMICOLON ";"
4:1 RBRACE "}"
5:1 END_OF_FILE ""
````
### Запуск парсера

#### Для запуска синтаксического анализатора:

```bash
compiler parse --input <путь_к_входному_файлу> --output <путь_к_выходному_файлу> [--format {text|dot|json}] [--verbose]
Параметры:

--input, -i — путь к файлу с исходным кодом

--output, -o — путь к выходному файлу

--format, -f — формат вывода AST (по умолчанию: text)

--verbose, -v — подробный вывод (включая ошибки парсинга)
````
Примеры:

```bash
# Текстовый формат
compiler parse -i examples/factorial.src -o ast.txt

# Graphviz DOT
compiler parse -i examples/factorial.src -f dot -o ast.dot

# JSON
compiler parse -i examples/factorial.src -f json -o ast.json

# Подробный вывод
compiler parse -i examples/factorial.src -o ast.txt -v
Форматы вывода AST
Текстовый формат
Человеко-читаемое представление с отступами и аннотациями строк.


Для визуализации графа AST. Преобразование в PNG:
````
```bash
dot -Tpng ast.dot -o ast.png
````

#### Структура тестов


### Добавление новых тестов

Создайте файл с исходным кодом в соответствующей директории (valid или invalid) с расширением .src. Имя файла должно отражать суть теста (например, test_long_identifier.src).

Запустите лексер на этом файле вручную и убедитесь, что вывод корректен.

Сохраните этот вывод в файл с тем же именем, но с расширением .expected в той же директории.

При следующем запуске test_runner.py новый тест будет автоматически обнаружен и выполнен.
